# 页面乱码原因分析（角色管理、岗位管理等）

## 一、现象

- **角色管理**（`/system/role`）：页面中文显示为 `��ɫ����`、`Ȩ���ַ�`、`����`、`�޸�`、`ɾ��` 等乱码。
- **岗位管理**（`/system/post`）：页面中文显示为 `宀椾綅缂栫爜`、`璇疯緭鍏ュ矖浣嶇紪鐮�`、`鎼滅储`、`鏂板��` 等乱码。

## 二、根本原因：源文件编码不一致

### 1. 同一项目存在多份副本，编码不同

项目中存在多份代码副本，例如：

- `D:\AI\StuCourceSys\SchoolCourseSlecetSys\ruoyi-ui\` 
- `D:\AI\SchoolCourseSlecetSys\ruoyi-ui\`

同一 Vue 文件在不同副本中的**磁盘编码**可能不同：

- 一份是 **UTF-8**（显示正常）
- 另一份是 **GBK/GB2312**，或被以错误编码保存/复制（显示乱码）

Vite 构建时按“当前打开的这份文件”的字节来打包。若构建的是**非 UTF-8** 的那份，打出来的 JS/HTML 里的中文就会是乱码。

### 2. 两类典型乱码形态

| 形态 | 示例 | 含义 |
|------|------|------|
| **UTF-8 被当作 Latin-1/西欧** | `��ɫ����`、`����`、`�޸�` | 文件实际是 UTF-8，但被某环节按单字节（如 Windows-1252）解析，多字节被拆成 �。 |
| **UTF-8 被当作 GBK** | `宀椾綅缂栫爜`、`鎼滅储`、`鏂板��` | 文件是 UTF-8，但被按 GBK 解码（或相反：GBK 被当 UTF-8），导致错位成“怪字”。 |

因此：**乱码来自“源文件本身的字节”与“构建/运行时所采用的编码”不一致**，而不是来自后端接口或数据库。

### 3. 可能加剧编码问题的环节

- **编辑器默认编码**：若 IDE/编辑器默认是 GBK，保存时会把 UTF-8 转成 GBK，或保存成“错误 UTF-8”。
- **复制/迁移**：从别处复制文件或整目录时未按“二进制”或“UTF-8”处理，导致编码被转换。
- **运行目录与修改目录不一致**：修改的是 A 路径下的文件，但 `npm run dev` / `npm run build` 在 B 路径执行，构建的是另一份“乱码源文件”。

## 三、为何“改完又乱码”

- 用“正确 UTF-8 内容”覆盖了**其中一个副本**（如 StuCourceSys 下的 `role/index.vue`），但：
  - 实际运行/打包的是**另一副本**（如 SchoolCourseSlecetSys），或
  - 覆盖后又被编辑器/工具以 **GBK 或“带 BOM 的 UTF-8”** 再次保存，编码再次被改坏。
- 未统一约定：**所有 .vue/.js 源文件一律使用 UTF-8（无 BOM）保存**。

## 四、解决思路

### 1. 统一源文件编码（治本）

- **约定**：所有前端源文件（`.vue`、`.js`、`.ts`、`.json` 等）统一为 **UTF-8 无 BOM**。
- **编辑器**：在 VS Code/Cursor 中设置  
  `"files.encoding": "utf8"`  
  并对已有文件执行“通过编码重新打开”或“以 UTF-8 保存”。
- **只保留一份前端工程**：避免多份副本；若必须多份，用同一份“已修正为 UTF-8”的目录作为唯一源，其余仅作只读备份或通过 Git 同步。

### 2. 用“正确版本”覆盖乱码文件（治标）

- 以**已知显示正常**的那份（如 `SchoolCourseSlecetSys` 下的 `role/index.vue`、`post/index.vue`）为模板。
- 用其**完整内容**覆盖乱码副本（如 `StuCourceSys` 下同路径文件），并**确保保存为 UTF-8 无 BOM**。
- 覆盖后在该目录执行一次 `npm run build` 或 `npm run dev`，确认页面中文正常。

### 3. 确认构建与运行路径

- 执行 `npm run dev` / `npm run build` 的目录，必须是您修改并保存为 UTF-8 的那份前端工程。
- 若使用 `D:\AI\StuCourceSys\SchoolCourseSlecetSys\ruoyi-ui`，则角色/岗位的 `index.vue` 应在此路径下修正并保存为 UTF-8。

### 4. 页面与构建配置已排除

- `index.html` 中已有 `<meta charset="utf-8">`，页面声明正确。
- Vite 默认按 UTF-8 读源文件；问题在于**源文件在磁盘上不是 UTF-8**，而不是 Vite 配置错误。

## 五、小结

| 项目 | 说明 |
|------|------|
| **原因** | 角色/岗位等页面的 Vue 源文件在部分副本中**不是 UTF-8**，或曾被以 GBK/错误编码保存；构建时把这些错误字节打进 bundle，导致浏览器显示乱码。 |
| **为何反复出现** | 多副本导致改 A 未改 B；或修正后再次被编辑器/工具以非 UTF-8 保存。 |
| **根本办法** | 统一全部源文件为 UTF-8 无 BOM，并只维护一份前端工程（或严格用同一份作为构建源）。 |
| **立即修复** | 用“正确中文内容”的版本覆盖乱码文件，并确保保存为 UTF-8；在**同一目录**下重新构建并强刷浏览器（Ctrl+F5）验证。 |

当前已对 **角色管理**、**岗位管理** 的 `index.vue` 用正确 UTF-8 内容覆盖 `StuCourceSys` 下对应文件；请在 **该目录** 下重新执行 `npm run dev` 或 `npm run build` 并验证页面显示。
