# 学生删除后重新导入问题 - 解决方案

## 问题现象

删除学生后重新导入时报错：
```
导入失败！共 290 条数据格式不正确：
1、学号 3101 已存在（包含物理冲突或账号冲突）
2、学号 3102 已存在（包含物理冲突或账号冲突）
...
```

## 根本原因

**删除操作使用了逻辑删除而不是物理删除！**

数据库检查结果：
- ✅ 用户表：0 条记录（已物理删除）
- ❌ 学生表：290 条逻辑删除记录（`del_flag = '2'`）

这导致：
1. 学生表中的学号仍然存在（虽然 `del_flag = '2'`）
2. 学号唯一约束仍然生效
3. 重新导入时检测到学号已存在

## 为什么会出现逻辑删除？

可能的原因：
1. **前端缓存** - 浏览器缓存了旧版本的前端代码
2. **代码未重新编译** - 后端代码修改后没有重新编译
3. **其他删除入口** - 可能有其他地方调用了逻辑删除

## 已执行的修复操作

### 1. 清理数据库
```sql
-- 清理学生表中的逻辑删除记录
DELETE FROM stu_student_info WHERE del_flag = '2';  -- 删除了 290 条

-- 清理用户表中的逻辑删除记录
DELETE FROM sys_user WHERE user_type = '01' AND del_flag = '2';  -- 删除了 0 条
```

### 2. 验证清理结果
```
学生表记录数: 0
用户表记录数: 0
```

数据库已完全清理干净！

这是物理删除，没有问题。

### Service 实现（正确）
```java
@Override
@Transactional(rollbackFor = Exception.class)
public int deleteStudentById(Long id)
{
    // 1. 物理删除学生记录
    studentMapper.deleteStudentById(id);
    
    // 2. 物理删除用户记录
    if (student.getUserId() != null) {
        userRoleMapper.deleteUserRoleByUserId(student.getUserId());
        userPostMapper.deleteUserPostByUserId(student.getUserId());
        userMapper.deleteUserByIdPhysical(student.getUserId());
    }
    return 1;
}
```
这是物理删除，没有问题。

### Controller（正确）
```java
@DeleteMapping("/{id}")
public AjaxResult remove(@PathVariable Long id)
{
    return toAjax(studentService.deleteStudentById(id));
}
```
调用的是物理删除方法，没有问题。

### 前端 API（正确）

调用的是正确的 DELETE 接口，没有问题。

## 问题分析

代码层面都是正确的物理删除，但是数据库中却出现了逻辑删除的记录。这说明：

**可能是使用了旧版本的代码或缓存！**

## 预防措施

### 1. 定期清理逻辑删除记录

创建定时任务或手动执行：
```sql
-- 清理学生表
DELETE FROM stu_student_info WHERE del_flag = '2';

-- 清理用户表
DELETE FROM sys_user WHERE user_type = '01' AND del_flag = '2';
```

### 2. 监控删除操作

在 `deleteStudentById` 方法中添加日志：
```java
log.info("物理删除学生：id={}, studentNo={}", id, student.getStudentNo());
```

### 3. 数据库约束优化

考虑修改唯一约束，使其只对活跃记录生效：
```sql
-- 删除旧约束
ALTER TABLE stu_student_info DROP CONSTRAINT stu_student_info_student_no_key;

-- 创建部分唯一索引（只对 del_flag='0' 的记录生效）
CREATE UNIQUE INDEX stu_student_info_student_no_active_key 
ON stu_student_info (student_no) 
WHERE del_flag = '0';
```

但是由于我们已经改为物理删除，这个优化不是必需的。
