# 学生删除后重新导入问题修复

## 问题描述

删除学生后重新导入相同学号的学生时报错：
```
学号 3101 已存在（包含物理冲突或账号冲突）
```

## 问题原因

1. 删除学生时，用户表（sys_user）使用的是**逻辑删除**（软删除）
2. 逻辑删除只是将 `del_flag` 设置为 '2'，记录仍然存在
3. 重新导入时，系统检查用户名（学号）是否存在，包括已删除的记录
4. 导致学号冲突，无法重新导入

## 解决方案

修改学生删除逻辑，对关联的用户账号进行**物理删除**（真正删除记录），而不是逻辑删除。

## 修改内容

### 1. StudentServiceImpl.java

**添加 Mapper 依赖**：
```java
@Autowired
private com.ruoyi.system.mapper.SysUserMapper userMapper;

@Autowired
private com.ruoyi.system.mapper.SysUserRoleMapper userRoleMapper;

@Autowired
private com.ruoyi.system.mapper.SysUserPostMapper userPostMapper;
```

**修改删除方法**：
```java
@Override
@Transactional(rollbackFor = Exception.class)
public int deleteStudentById(Long id)
{
    StuStudentInfo student = studentMapper.selectStudentById(id);
    if (student == null)
    {
        return 0;
    }
    // 先删除学生记录
    studentMapper.deleteStudentById(id);
    // 物理删除用户记录，避免学号冲突
    if (student.getUserId() != null)
    {
        // 删除用户与角色关联
        userRoleMapper.deleteUserRoleByUserId(student.getUserId());
        // 删除用户与岗位关联
        userPostMapper.deleteUserPostByUserId(student.getUserId());
        // 物理删除用户记录（直接DELETE，而不是逻辑删除）
        userMapper.deleteUserByIdPhysical(student.getUserId());
    }
    return 1;
}
```

### 2. SysUserMapper.java

**添加物理删除方法**：
```java
/**
 * 物理删除用户（用于学生删除后允许重新导入）
 * 
 * @param userId 用户ID
 * @return 结果
 */
public int deleteUserByIdPhysical(Long userId);
```

### 3. SysUserMapper.xml

**添加物理删除 SQL**：
```xml
<delete id="deleteUserByIdPhysical" parameterType="Long">
    delete from sys_user where user_id = #{userId}
</delete>
```

## 修改后的效果

1. ✅ 删除学生时，关联的用户账号会被物理删除
2. ✅ 删除后可以重新导入相同学号的学生
3. ✅ 不影响其他用户的逻辑删除功能
4. ✅ 保持事务一致性

## 测试步骤

1. 删除一个或多个学生
2. 重新导入相同学号的学生
3. 验证导入成功，没有"学号已存在"的错误

## 注意事项

1. 此修改只影响学生删除，不影响其他用户（教师、管理员等）的删除
2. 其他用户仍然使用逻辑删除（软删除）
3. 学生删除后，相关的用户账号会被永久删除，无法恢复
4. 建议在删除前做好数据备份

修改完成后，需要重新编译后端：

```powershell
# 在 SchoolCourseSlecetSys 目录下运行
mvn clean package -DskipTests
```

```powershell
mvn package -DskipTests
```
